Документация проекта RSK_back
1. Общая архитектура

Проект RSK_back представляет собой backend-систему, построенную на микросервисной архитектуре. Каждый микросервис изолирован, имеет собственную базу данных и управляется через отдельный docker-compose.yml. Коммуникация между сервисами происходит через сеть Docker-контейнеров (по hostname) и асинхронные сообщения через RabbitMQ, HTTPX

1.1. Структура проекта

```bash
RSK_back/
├── docker-compose.yml           # Основной файл для production-окружения
├── docker-compose.override.yml  # Файл для локальной разработки (переопределяет конфигурацию)
├── monitoring/                  # Конфигурации для мониторинга (Prometheus)
├── backups/                     # Директория с бекапами баз данных
├── admin-panel_service/         # Сервис админ-панели (клиентская часть)
├── admin_service/               # Telegram бот для администраторов
├── auth_service/                # Сервис аутентификации и управления пользователями
├── learning_service/            # Сервис управления обучением (курсы, задания)
├── orgs_service/                # Сервис организаций
├── projects_service/            # Сервис управления проектами ("Звезда")
├── teams_service/               # Сервис управления командами
├── user_profile/                # Сервис пользовательских профилей
```

2. Быстрый старт
2.1. Предварительные требования

    Установленный Docker и Docker Compose

    Git

2.2. Клонирование репозитория
```bash
git clone <url-репозитория> RSK_back
```
```bash
cd RSK_back
```
2.3. Настройка окружения
в каждом сервисе создайте .env файлы пример auth_service/.env

Важно: Отредактируйте каждый .env файл, указав корректные пароли и секретные ключи. ( можно смотреть в конфигах пример auth_service/app/config.py и там будет список того что должно быть в этом сервисе в .env ) 

2.4. Запуск
Для production (основной docker-compose файл):
```bash
docker-compose -f docker-compose.yml up -d --build
```
Этот файл использует production-конфигурации и сети.
Для локальной разработки и тестирования:
```bash
docker-compose -f docker-compose.override.yml up -d --build
```
Этот файл может содержать дополнительные сервисы для отладки или измененные настройки для удобной разработки (например, проброс портов на хост).
3. Описание микросервисов
3.1. Auth Service (auth_service)

Назначение: Центральный сервис аутентификации, регистрации и управления пользователями.
Стек: FastAPI, SQLAlchemy, Alembic, PostgreSQL, RabbitMQ, JWT, OAuth2 (VK, Yandex).
Основные функции:

    Регистрация пользователя с подтверждением email.

    Аутентификация по логину/паролю, а также через VK и Яндекс.

    Управление JWT-токенами и установка HttpOnly cookies.

    Сброс пароля через email.

    Публикация событий о создании и верификации пользователя в RabbitMQ.

    Административные функции: просмотр всех пользователей, удаление пользователя.

Ключевые компоненты:

    Роуты: /users_interaction — содержит все эндпоинты аутентификации и управления.

    CRUD: UserCRUD (см. auth_service/app/cruds/users_crud/crud.py) — инкапсулирует логику работы с моделью пользователя.

    Модели: User в auth_service/app/db/models/user.py.

Основные эндпоинты:

    POST /users_interaction/register/ — регистрация нового пользователя.

    POST /users_interaction/login/ — вход в систему.

    POST /users_interaction/logout/ — выход.

    GET /users_interaction/confirm-email — подтверждение email по токену.

    GET /users_interaction/get_users/ — получить всех пользователей (требуются права админа).

3.2. Learning Service (learning_service)

Назначение: Управление обучающими курсами, заданиями и прогрессом пользователей.
Стек: FastAPI, SQLAlchemy, Alembic, PostgreSQL, RabbitMQ, Celery, Redis.
Основные функции:

    CRUD для курсов (доступно админам).

    Отслеживание прогресса пользователя по курсам.

    Прием и проверка заданий (сабмитов) пользователей.

    Распределение заданий между модераторами: система автоматически назначает до 5 заданий на модератора. Если у модератора нет заданий, он получает новые. Задания "блокируются" на время проверки (10 минут).

    Отправка уведомлений о результате проверки на email пользователя.

    Интеграция с Celery для фоновых задач.

Ключевые компоненты:

    Роуты:

        /courses — управление курсами.

        /submissions — управление сабмитами.

        /moderator-assignments — получение и проверка назначенных заданий.

    CRUD:

        course_crud — логика работы с курсами.

        submission_crud — логика работы с заданиями (см. learning_service/app/crud/submission_crud/crud.py).

        user_progress_crud — логика работы с прогрессом.

3.3. Orgs Service (orgs_service)

Назначение: Управление организациями (школы, вузы, партнеры). Использует данные из официального реестра (DaData).
Стек: FastAPI, SQLAlchemy, Alembic, PostgreSQL, DaData API.
Основные функции:

    Поиск организации по ИНН через DaData API.

    CRUD для организаций.

    Получение списка организаций с фильтрацией по региону и названию, сортировкой.

    Подсчет количества участников и команд в организации через интеграцию с user_profile и teams_service.

Ключевые компоненты:

    Роуты: /organizations — все эндпоинты организаций.

    CRUD: OrgsCRUD (см. orgs_service/app/cruds/orgs_crud.py).

    Импорт: import_excel_to_sql для загрузки готовых списков организаций из Excel.

Основные эндпоинты:

    GET /organizations/all — получить список организаций (поддерживает region, name, sort_by, order, limit, offset).

    GET /organizations/org/{org_id} — получить детальную информацию об организации.

    POST /organizations/create — создать организацию по ИНН.

3.4. Projects Service (projects_service)

Назначение: Управление проектами "Звезда" и связанными с ними задачами.
Стек: FastAPI, SQLAlchemy, Alembic, PostgreSQL.
Основные функции:

    CRUD для проектов и задач внутри проектов.

    Механизм "старта" задачи командой (только лидер команды).

    Механизм отправки решения задачи (сабмита) на проверку.

    Распределение сабмитов для модераторов: аналогично learning service, но с использованием собственной логики блокировок. Модератору выдаются новые сабмиты только после того, как он обработает уже взятые.

Ключевые компоненты:

    Роуты: /zvezda — все эндпоинты проектов.

    CRUD: ZvezdaCRUD (см. projects_service/app/cruds/crud.py).

    Клиенты: TeamsClient для взаимодействия с teams_service и получения информации о командах.

Основные эндпоинты:

    GET /zvezda/projects — список проектов.

    POST /zvezda/projects/{project_id}/tasks — создать задачу в проекте.

    POST /zvezda/tasks/{task_id}/start — начать выполнение задачи командой.

    POST /zvezda/tasks/{task_id}/submit — отправить решение задачи.

    GET /zvezda/moderator/tasks — получить список заданий для проверки (для модератора).

    POST /zvezda/moderator/{submission_id}/review — проверить задание (принять/отклонить). Принятое задание начисляет очки команде.

3.5. Teams Service (teams_service)

Назначение: Управление командами и их составом.
Стек: FastAPI, SQLAlchemy, Alembic, PostgreSQL.
Основные функции:

    Создание команды (требуется быть лидером и не состоять в другой команде).

    Вступление и выход из команды.

    Ограничения по составу: В команде может быть 1 студент и 3 учителя. Логика проверки реализована в TeamCRUD.analyze_team_composition и TeamCRUD.can_user_join_team.

    Получение информации о команде, ее участниках и их профилях.

    Начисление баллов команде (через вызов add_points_to_team из projects_service).

Ключевые компоненты:

    Роуты: /teams (с вложенными роутерами для управления, членства и поиска).

    CRUD: TeamCRUD (см. teams_service/app/cruds/teams_crud/crud.py).

    Клиенты: UserProfileClient, OrgsClient для получения профилей и данных об организациях.

Основные эндпоинты:

    POST /teams/register — создать команду (требуется токен).

    POST /teams/join_team/{team_id} — вступить в команду.

    GET /teams/team_members/{team_id} — получить участников команды.

    GET /teams/get_team_by_id/{team_id} — детальная информация о команде (состав, свободные места).

    PATCH /teams/update_team_data/{team_id} — обновить данные команды.

3.6. User Profile Service (user_profile)

Назначение: Хранение и управление расширенной информацией о пользователе.
Стек: FastAPI, SQLAlchemy, Alembic, PostgreSQL, RabbitMQ.
Основные функции:

    Хранение профиля пользователя (ФИО, регион, организация, роль).

    Обновление профиля самим пользователем.

    Изменение роли пользователя: Обновление роли публикуется в RabbitMQ (user_role_updated), чтобы другие сервисы (например, teams) могли обновить кэш ролей.

    Пакетное получение профилей пользователей по списку ID.

    Подсчет количества участников в организации.

    Отслеживание, к какой команде и организации привязан пользователь.

Ключевые компоненты:

    Роуты: /profile_interaction (с вложенными роутерами для управления профилем, пакетных операций и административных действий).

    CRUD: ProfileCRUD (см. user_profile/app/cruds/profile_crud.py).

    Клиенты: OrgsClient для получения данных об организации при отображении профиля.

Основные эндпоинты:

    GET /profile_interaction/get_my_profile/ — получить свой профиль.

    PATCH /profile_interaction/update_my_profile/ — обновить свой профиль.

    PATCH /profile_interaction/my-role — изменить свою роль (на student/teacher).

    PATCH /profile_interaction/admin-role — изменить роль другого пользователя (на admin/moder, требует админских прав).

    POST /profile_interaction/get_users_batch — получить данные пользователей по списку ID.

3.7. Admin Service (admin_service)

Назначение: Telegram-бот для администраторов системы.
Стек: Python, Aiogram.
Основные функции:

    Уведомление администраторов о создании команд с несуществующей организацией.

    Потенциальное расширение для других административных функций.

3.8. Admin Panel Service (admin-panel_service)

Назначение: Внутренний клиент для админ-панели, вероятно, для взаимодействия с другими сервисами.
Стек: Python, возможно, FastAPI.
4. Взаимодействие между сервисами
4.1. Синхронное взаимодействие (HTTP)

Сервисы общаются друг с другом по HTTP, используя hostname контейнеров в Docker-сети. Например, auth_service обращается к user_profile через http://user_profile:8000. Клиенты для таких запросов находятся в папке services каждого сервиса (например, auth_client.py, teams_client.py).
4.2. Асинхронное взаимодействие (RabbitMQ)

Используется для публикации событий, которые не требуют немедленного ответа.

    Очередь: user_events

    События:

        user_registered: Публикуется auth_service при регистрации.

        user_verified: Публикуется auth_service после подтверждения email.

        user_role_updated: Публикуется user_profile после изменения роли пользователя.

    Потребители:

        auth_service имеет role_consumer, который слушает события обновления ролей.

        Другие сервисы могут подписываться на эти события для обновления своего локального состояния или кэша.

5. Базы данных

Каждый сервис, который хранит состояние, имеет собственную базу данных PostgreSQL. Миграции выполняются с помощью Alembic внутри каждого сервиса при запуске ( смотреть докерфайлы каждого сервиса и общий компоуз файл).

6. Мониторинг

Проект интегрирован с Prometheus. Метрики (например, количество активных пользователей в auth_service) доступны по эндпоинту /metrics . Конфигурация Prometheus находится в monitoring/prometheus.yml.

7. Безопасность

    JWT-токены хранятся в HttpOnly cookies для защиты от XSS.

    Права доступа: Эндпоинты защищены депенденсами (get_current_user, get_moderator, get_admin), которые проверяют JWT и роль пользователя.

    CORS: Настроен через domain=".rosdk.ru" в cookies.

8. Резервное копирование (Backups)

Директория backups/ содержит дампы баз данных всех сервисов. Это позволяет восстановить данные в случае сбоя.
